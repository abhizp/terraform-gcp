Network Peering:
-- Classic VPN
-- HA VPN -- regional service, and its limited to one VPC

Cloud Interconnect: -- layer 2 -- transport layer, access to private IP
-- layer2 vs layer3
-- dedicated vs shared 
-- direct interconnect -- cloud to on-prem with routing facility in google's location
-- shared interconnect -- isp partner will help interconnect on-prem network to cloud, physical network connection required.
-- cloud interconnect -- cloud to cloud interconnect, routers, bgp and many more required.
-- VPN -- also present -- just very less throughput 

Peering: access to only public IP of google
-- same as interconnect but instead of dedicated direct cables, hardware and network component, it uses google's infra and all.
-- peering is limited to within cloud, i.e; vpc to vpc only, and not on-prem to cloud like interconnect.
-- peering uses cloud-providers network backbone and does not need any special network components setup except for routers

direct peering: used to access Google api and workspace and not the VPC, uses BGP and happens at one goole's edge location.
carrier peering : uses ISP to connect to google's apps and services
NO SLA for peering -- layer 3 - network layer

Network load balancer:
-- handles tcp traffic

Proxy and pass-through load balancers differ primarily in how they handle connections: 
-- proxy load balancers terminate client connections and create new ones to backends (offering high intelligence/security), 
-- pass-through load balancers (L4) directly forward packets to backends, preserving client IP addresses for maximum performance.
   -- direct server return is done in pass through, packets from server side do not go to load balancer again, they go to client directly.

SLI,SLO, SLA:

SLI: Service level indicators, or SLIs, are carefully selected monitoring metrics that measure one aspect of a service's reliability.

-- it is expressed as the ratio of two numbers: the number of good events divided by the count of all valid events.
-- Service Level Indicators (SLIs) measure the actual performance of a service (e.g., latency, uptime)
-- SLIs provide the raw data ("we are 99.9% available"),

SLO: Service level objective
-- while Service Level Objectives (SLOs) are the internal, targeted benchmarks for those metrics
-- SLOs define the acceptable range ("we aim for 99.95% availability") to manage error budgets.
-- SLOs define the goal for service reliability.
-- eg: 99.9% of HTTP requests must be successful over a 30-day period.
-- SLO, combines a service level indicator with a target reliability.
-- SLOs that are S.M.A.R.T. SLOs should be specific.
-- S,M,A,R,T -- Specific, Measureable, Achievable, Relevant, Time Bound.

SLA: Service Level Agreement
-- which are commitments made to your customers that your systems and applications will have only a certain amount of “down time.”
-- SLA describes the minimum levels of service that you promise to provide to your customers and what happens when you break that promise.

gcloud <group> <sub-group> <action> <resource_name> --<flag>=<value>

[Tool] [Service] [Resource] [Action] --[Flags]
Tool: Always gcloud.
Service (Group): The GCP service (e.g., compute, container, storage).
Resource (Subgroup): The specific item within that service (e.g., instances, clusters, buckets).
Action: What you want to do (e.g., create, list, describe, delete).
Flags: Key-value pairs that modify the command (e.g., --project, --zone, --format).
